# Convex 튜토리얼: 앱 확장하기

Convex는 처음부터 확장을 위해 설계되었습니다. 이전 섹션에서 우리는 이미 액션을 작게 유지하고 대부분의 로직을 쿼리와 뮤테이션에 유지하는 것이 빠르고 확장 가능한 백엔드를 구축하는 데 중요하다는 것을 이야기했습니다.

앱을 빠르고 확장 가능하게 유지하는 몇 가지 다른 방법에 대해 이야기해 보겠습니다.

## 인덱싱된 쿼리

인덱스는 데이터베이스에 조회 구조를 만들어 데이터 필터링을 매우 빠르게 만듭니다. 채팅 앱에서 한 사용자의 `messages`만 조회하는 방법을 만들고 싶다면, Convex에 `messages` 테이블의 `user` 필드를 인덱싱하도록 지시하고 `withIndex` 구문으로 쿼리를 작성합니다.

[인덱스 사용 방법을 배우세요](/database/reading-data/indexes/indexes.md).

## 동일한 문서에 대한 너무 많은 쓰기

앱에 카운터를 표시하기로 결정했다고 가정해 보겠습니다. 숫자 필드를 읽고, 1을 더하고, 데이터베이스의 동일한 필드를 업데이트하는 뮤테이션을 작성할 수 있습니다. 어느 시점에서, 이 패턴은 [낙관적 동시성 제어 충돌](/error#1)을 일으킬 수 있습니다. 이것은 데이터베이스가 그렇게 빠르게 문서를 업데이트할 수 없다는 것을 의미합니다. 모든 데이터베이스는 이런 종류의 패턴에 문제가 있습니다.

샤딩된 카운터라는 것을 구축하는 것을 포함하여 [이를 처리하는 몇 가지 방법](/error#remediation)이 있습니다...

하지만 직접 고급 확장 기술을 배우기 전에, Convex 컴포넌트를 사용하는 더 나은 방법이 있습니다.

## Convex Components로 확장 모범 사례

위의 카운터의 경우, Convex 팀은 이미 사용할 수 있는 [확장 가능한 카운터](https://www.convex.dev/components/sharded-counter) Convex 컴포넌트를 만들었습니다.

Convex 컴포넌트는 npm 라이브러리로 Convex 백엔드에 설치됩니다. 샌드박스 처리되어 있어 명시적으로 제공되지 않는 한 앱의 테이블을 읽거나 앱의 함수를 호출할 수 없습니다.

AI 에이전트 [워크플로우](https://www.convex.dev/components/workflow), [리더보드](https://www.convex.dev/components/aggregate), [기능 플래그](https://www.convex.dev/components/launchdarkly) 또는 [속도 제한기](https://www.convex.dev/components/rate-limiter)와 같은 더 복잡한 기능을 구축할 때, 이 문제를 해결하는 Convex 컴포넌트가 이미 있을 수 있습니다.

## 마무리

이 튜토리얼에서 많은 것을 다뤘습니다. Convex 동기화 엔진의 기본 구성 요소를 형성하는 쿼리, 뮤테이션 및 데이터베이스로 [채팅 앱 구축](/tutorial/index.mdx)으로 시작했습니다. 그런 다음 스케줄러를 사용하여 작업을 조정하면서 백엔드에서 [외부 API를 호출](/tutorial/actions.mdx)했습니다. 마지막으로 [Convex 컴포넌트](https://www.convex.dev/components)가 깔끔한 패키지로 확장 모범 사례를 제공한다는 것을 배웠습니다.

더 많은 팁을 찾고 있다면, [모범 사례](/understanding/best-practices/best-practices.mdx)를 읽고 [커뮤니티](https://www.convex.dev/community)에 참여하세요.

Convex를 사용하면 MVP를 빠르게 구축한 다음 새로운 높이로 확장할 수 있습니다. 많은 훌륭한 제품이 이미 그렇게 했습니다. 좋은 동료들과 함께하고 있습니다.
